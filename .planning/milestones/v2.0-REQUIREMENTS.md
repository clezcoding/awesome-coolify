# Requirements: Coolify MCP Server — v2.0 Creation & CRUD

**Defined:** 2026-07-16
**Core Value:** AI agent manages Coolify instances — deploy, logs, diagnose, and **create infrastructure from scratch** — via one MCP server.

**Research basis:** `.planning/research/SUMMARY.md` (Context7 + Coolify API 4.1.x feasibility verified 2026-07-16)

## v2.0 Requirements

### Private Keys (KEY)

- [ ] **KEY-01**: Agent can list private keys
- [ ] **KEY-02**: Agent can get private key details by UUID
- [ ] **KEY-03**: Agent can create private key (name, PEM content, description)
- [ ] **KEY-04**: Agent can update private key metadata
- [ ] **KEY-05**: Agent can delete private key with `confirm: true`

### Servers (SRV)

- [ ] **SRV-01**: Agent can create server (name, IP, port, user, private_key_uuid)
- [ ] **SRV-02**: Agent can update server configuration
- [ ] **SRV-03**: Agent can delete server with `confirm: true` and safe defaults (`delete_volumes=false`)
- [ ] **SRV-04**: Agent can trigger server validation (SSH/reachability) after create or on demand
- [ ] **SRV-05**: Agent can mark server as build server via update

### Projects & Environments (PROJ)

- [x] **PROJ-01**: Agent can create project (name, description)
- [x] **PROJ-02**: Agent can update project
- [x] **PROJ-03**: Agent can delete project with `confirm: true`
- [x] **PROJ-04**: Agent can create environment in project (name)
- [x] **PROJ-05**: Agent can delete environment with `confirm: true` (empty environment only)

### Applications (APP)

- [x] **APP-12**: Agent can create application from public Git (nixpacks/static/dockerfile/dockercompose build packs)
- [x] **APP-13**: Agent can create application from private Git via deploy key
- [x] **APP-14**: Agent can create application from private Git via GitHub App
- [x] **APP-15**: Agent can create application from inline Dockerfile
- [x] **APP-16**: Agent can create application from Docker registry image
- [x] **APP-17**: Agent can update application configuration (FQDN, build commands, health checks, labels, etc.)
- [x] **APP-18**: Agent can delete application with `confirm: true` and safe defaults (`delete_volumes=false`)
- [x] **APP-19**: Agent can configure HTTP basic auth on application
- [x] **APP-20**: Agent can enable instant deploy on create
- [x] **APP-21**: Agent receives structured recovery hints on domain conflict (409) with optional `force_domain_override`

### Services (SVC)

- [x] **SVC-06**: Agent can create one-click service by type
- [x] **SVC-07**: Agent can create custom service from Docker Compose YAML (MCP accepts raw YAML, encodes base64 for API)
- [x] **SVC-08**: Agent can update service configuration and compose
- [x] **SVC-09**: Agent can delete service with `confirm: true` and safe defaults
- [x] **SVC-10**: Agent receives structured recovery hints on domain conflict with optional `force_domain_override`

### Databases (DB)

- [x] **DB-01**: Agent can create database (postgresql, mysql, mariadb, mongodb, redis, clickhouse, dragonfly, keydb)
- [x] **DB-02**: Agent can update database configuration
- [x] **DB-03**: Agent can delete database with `confirm: true` and safe defaults (`delete_volumes=false`)
- [x] **DB-04**: Agent can configure public access and public port on database

### Environment Variables (ENV)

- [x] **ENV-01**: Agent can create env var on application, service, or database
- [x] **ENV-02**: Agent can update env var by UUID
- [x] **ENV-03**: Agent can delete env var by UUID
- [x] **ENV-04**: Agent can bulk-update env vars on application (`PATCH /applications/{uuid}/envs/bulk`)
- [x] **ENV-05**: Agent can sync local `.env` file to application (MCP-side diff: create/update keys)
- [x] **ENV-06**: Env var flags supported: is_preview, is_literal, is_multiline, is_shown_once

### Database Backups (BAK)

- [x] **BAK-01**: Agent can create backup schedule for database (frequency, retention, S3 optional)
- [x] **BAK-02**: Agent can list backup configurations for database
- [x] **BAK-03**: Agent can update backup configuration
- [x] **BAK-04**: Agent can delete backup configuration with `confirm: true`
- [x] **BAK-05**: Agent can trigger immediate backup (`backup_now`)
- [x] **BAK-06**: Agent can list backup execution history

### Safety & DX (SAF)

- [x] **SAF-01**: All delete actions require `confirm: true` or return `COOLIFY_CONFIRM_REQUIRED`
- [x] **SAF-02**: Delete handlers default `delete_volumes=false`, `delete_configurations=false`, `docker_cleanup=false` unless explicitly overridden
- [x] **SAF-03**: Create/update payloads validated via Zod before API call
- [x] **SAF-04**: Secrets in create/update responses masked by default; `reveal: true` opt-in unchanged from v1

## Future Requirements (v2.x / v1.1)

Deferred — not in v2.0 roadmap phases.

### Multi-Instance (CTX)

- **CTX-04**: Agent can manage multiple instances in `~/.coolify-mcp/instances.json`
- **CTX-05**: Agent can set default instance and switch between instances
- **CTX-06**: Agent can override API token per request

### Logs (v1.1)

- **SVC-04**: Agent can read service/database bounded logs (Coolify API gap in 4.1.x)

### Distribution

- **DIST-04**: Live npm publish of `awesome-coolify-mcp` (gated on v2 feature-complete)

### Teams, Cloud, GitHub, Observability (v2.x+)

See [milestones/v1.0-REQUIREMENTS.md](milestones/v1.0-REQUIREMENTS.md) § v2 for V2-TEAM, V2-CLOUD, V2-GH, V2-OBS, V2-DX, etc.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Storage Volume CRUD | **API gap** — no REST endpoints; volumes via compose/dockerfile only (Context7 verified) |
| Scheduled Tasks / Cron CRUD | **API gap** — no REST endpoints (Context7 verified) |
| Container shell exec / file upload | **API gap** — WebSocket/UI only in Coolify 4.1.x |
| Application Docker Compose create | **Deprecated** — route to `POST /services` instead |
| Multi-instance CRUD | Deferred to v2.x per milestone scope decision |
| SVC-04 service/DB logs | Separate v1.1 milestone |
| Execute command in container | Coolify 4.1.x API broken/missing (unchanged from v1) |
| npm publish | Gated until v2 feature-complete |
