<objective>
Gap closure for Phase 11 UAT gaps G-11-3 and G-11-4 — both share one root cause: `projectServiceCompose()` in `src/utils/yaml-validator.ts` only reads `docker_compose_raw` and assumes base64. Coolify 4.1.2 GET responses return **plain YAML** in `docker_compose_raw` (and `docker_compose`) for one-click and custom-compose services, so `decodeCompose` regex-rejects the string → `compose_decode_error` → no `compose` alias. Agent sees Coolify-native `docker_compose` instead of the MCP `compose` alias.

Purpose: Restore the D-06 contract (agent never sees base64; `compose` field always carries decoded YAML) against real Coolify 4.1.2 response shapes. Unblocks Phase 11 sign-off and prevents the same projection bug from leaking into Phase 12 env-var sync (which reads service compose for `.env` interpolation).

Output: Single-file fix to `projectServiceCompose` with a 3-step fallback chain (base64 decode → plain-YAML passthrough → `docker_compose` field), plus unit tests covering the live response shapes observed in UAT Tests 3 and 4. Live UAT re-test checkpoint for Tests 3+4.
</objective>

  <action>
    Rewrite `projectServiceCompose` in `src/utils/yaml-validator.ts` to apply a 3-step fallback chain before emitting an error:

    1. Read `docker_compose_raw` (string). If present and non-empty:
       a. Try `decodeCompose(docker_compose_raw)` — if it returns a non-null string AND `validateCompose(decoded).ok === true`, use `decoded` as the `compose` value.
       b. Else if `validateCompose(dockerComposeRaw).ok === true` (Coolify 4.1.2 plain-YAML shape — the raw string IS the YAML), use `dockerComposeRaw` directly as `compose`.
    2. If step 1 did not yield a `compose` value, read `docker_compose` (string). If present and non-empty AND `validateCompose(dockerCompose).ok === true`, use it as `compose`.
    3. Only if no `compose` source resolved: emit `compose_decode_error: 'compose field not decodable from docker_compose_raw or docker_compose'` (keep the existing error key so service.ts error handling stays compatible). This is the ONLY error message the function emits — no per-branch variants.

    When a `compose` value is resolved: build the result with `compose` set, then `delete result.docker_compose_raw` AND `delete result.docker_compose` (D-06 — strip both Coolify-native fields so the agent only sees the `compose` alias).

    When no `compose` value is resolved: strip BOTH `docker_compose_raw` AND `docker_compose` from the result unconditionally (single deterministic rule — no conditional keep-paths, no "inspect manually" escape hatch). The agent never sees Coolify-native compose fields under any branch; the `compose_decode_error` string carries the only failure signal.

    Reuse `decodeCompose` and `validateCompose` — do NOT introduce a new YAML parser or external dep. The `yaml` package `parse` already imported via `validateCompose` is the only YAML validator needed.

    Do NOT modify `encodeCompose`, `decodeCompose`, or `validateCompose` signatures — service.ts and database.ts depend on them.

    - `prefers plain-YAML docker_compose_raw over docker_compose when both plain` — input `{ uuid, docker_compose_raw: 'services:\n  web: nginx', docker_compose: 'services:\n  web: redis' }` → `result.compose` equals the nginx YAML.
    - `emits compose_decode_error only when no compose source resolves` — input `{ uuid, docker_compose_raw: '!!!garbage!!!', docker_compose: 'also : not : yaml : : :' }` (use a string that fails both base64 regex AND `validateCompose`) → `result.compose` undefined; `result.compose_decode_error` equals `'compose field not decodable from docker_compose_raw or docker_compose'`; both `result.docker_compose_raw` and `result.docker_compose` undefined (deterministic strip-both rule).
