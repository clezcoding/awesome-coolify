# Phase 19 — UI Design Contract

> Agent-facing design contract for the **Cursor MCP surface** (tool parameter panels, tool `description` catalogs, MCP prompt bodies, validation error copy). Not a React/web frontend. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |

## Spacing Scale

Mapped to **text density & structure** in tool descriptions and prompt bodies (Cursor host pixels out of control). Declared multiples of 4 for checker scoring:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px equiv. | Inline gaps inside catalog tokens — space after `·`; space after comma in param lists |
| sm | 8px equiv. | Blank line between tool title sentence and catalog line |
| md | 16px equiv. | Blank line between catalog block and safety footer |
| lg | 24px equiv. | Blank line between prompt intro sentence and numbered steps |
| xl | 32px equiv. | Section break before optional “Fallback / notes” block in prompts (max one per prompt) |
| 2xl | 48px equiv. | Reserved — do not use in Phase 19 prompt bodies (keeps ½–1 screen) |
| 3xl | 64px equiv. | Reserved — out of scope (playbooks → Phase 22) |

**Structural rules (concrete):**

| Rule | Value |
|------|-------|
| Catalog separator | single middle-dot ` · ` (U+00B7) with one space each side |
| Catalog max visual width | prefer ≤120 characters per logical line; if longer, continue on next line starting with `· ` (same catalog, no new heading) |
| Params per action in catalog | 1–3 key params only (D-05) |
| Optional params | suffix `?` on the param name: `force?`, `confirm?`, `lines?` |
| Prompt step count | 4–8 numbered steps (`1.` … `N.`) |
| Prompt body length | ~½–1 screen (~400–900 words max; prefer ≤600) |
| Safety footer | exactly one trailing line (or two short lines max) after catalog |

Exceptions: none beyond reserved 2xl/3xl unused.

## Typography

Roles map to **text hierarchy** in MCP descriptions/prompts (not CSS). Exactly four sizes + two weights:

| Role | Size | Weight | Line Height | Maps to |
|------|------|--------|-------------|---------|
| Body | 16px | 400 | 1.5 | Prompt step prose; tool one-line purpose sentence before catalog |
| Label | 14px | 400 | 1.4 | Safety footer; soft empty/missing-arg guidance; “optional” notes |
| Heading | 20px | 600 | 1.2 | Prompt title / first-line role (`Deploy application`, `Run diagnose scan`) — as markdown heading or bold lead if host allows plain text only, use ALL-CAPS-free Title Case lead sentence |
| Display | 28px | 600 | 1.2 | Unused in Phase 19 host (no custom chrome) — reserved token for checker; do not emit in strings |

**Mono emphasis (mental model, not font file):** wrap or present as bare identifiers — `application.deploy`, `uuid`, `confirm`, `deployment.watch` — never paraphrased into marketing synonyms.

**Weights used:** 400 (regular body/label), 600 (semibold heading only). No third weight.

## Color

Cursor host colors are uncontrolled. Contract = **semantic emphasis in copy/structure** using Mission Control 60/30/10:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0B1220` (ink metaphor) | Default prompt/tool prose — calm, neutral instructional voice |
| Secondary (30%) | `#1E293B` (panel metaphor) | Action catalog line; param lists; numbered step scaffolding |
| Accent (10%) | `#38BDF8` (signal metaphor) | Reserved for: (1) safety footer keywords `confirm`, `reveal`, `instance`; (2) forward-ref callout for `deployment.watch` in `deploy` prompt only |
| Destructive | `#F43F5E` (alert metaphor) | Destructive confirm messaging and catalog tokens that require `confirm?` / delete/stop/emergency wording |

Accent reserved for: safety footer keywords · `deployment.watch` forward-ref · never for decorative emphasis or every CTA.

**Text markers (implementation substitutes for color):**

| Emphasis | Marker |
|----------|--------|
| Safety / routing footer | Prefix `Safety:` then muted keywords |
| Destructive | Explicit verb + `confirm:true` requirement in same sentence |
| Optional args | Trailing `?` on param names in catalogs |
| Soft empty guidance | Prefix `Note:` (label weight) — never hard-fail language |

## Copywriting Contract

Language: **English only** for all tool descriptions, prompt bodies, and error strings (D-12).

### Primary CTAs (prompt lead verbs)

| Surface | Primary CTA |
|---------|-------------|
| Prompt `deploy` | Deploy application |
| Prompt `diagnose` | Run diagnose scan |
| Prompt `new-project` | Wire new project |
| Prompt `incident` | Triage incident |

### Tool description anatomy (every domain tool)

```
{One-sentence purpose.}
Actions: {action}({param}[, {param}?]) · {action}(...) · …
Safety: confirm for destructive ops · optional instance · reveal opt-in only
```

| Element | Copy |
|---------|------|
| Catalog format | `deploy(uuid, force?, confirm?) · logs(uuid, lines?) · …` (D-05; hand-maintained, D-06) |
| Catalog placement | Immediately after purpose sentence; label prefix `Actions:` |
| Safety footer | `Safety: confirm for destructive ops · optional instance · reveal opt-in only` (D-08) |
| Empty schema anti-pattern | Forbidden: empty `properties: {}`, top-level `oneOf`/`anyOf` (Pitfall 10 / DX-02) |

### Empty / missing-args (prompts — soft)

| Element | Copy |
|---------|------|
| Empty state heading | Missing arguments |
| Empty state body | If `uuid` (or other IDs) are omitted, resolve them from `.coolify/manifest.json` or ask the user before calling tools. Do not fail prompt load. |
| Soft guidance line | `Note: All prompt args are optional. Prefer manifest UUIDs when present; otherwise ask the user.` (D-10, D-14) |

### Error state (`COOLIFY_VALIDATION_ERROR`)

| Element | Copy |
|---------|------|
| Error state | Payload rejected by MCP Zod validation before any Coolify API call. Fix the listed fields, then retry the same tool with `action` set correctly. |
| recoveryHints (required shape) | Include at least: (1) selected `action` name in mono, (2) bullet/list of required fields for that action, (3) one next step (e.g. “Re-call `{tool}` with `{action}` and fields: …”). |
| Example hint pattern | `Action 'deploy' requires: uuid (or name|fqdn). Optional: force, wait, instance.` |

### Destructive confirmation

| Action class | Confirmation copy |
|--------------|-------------------|
| Resource delete | `Action 'delete' on {resource} '{uuid}' requires explicit confirmation. Retry with confirm:true after reviewing delete_preview.` |
| Env prune / sync apply | `Action 'envs:sync' requires confirm:true when applying (dry_run:false) or when prune:true.` |
| Emergency / bulk | Keep existing confirm-gate messages; reinforce via Safety footer — do not invent new confirm tokens. |

### Prompt body contracts (D-09, D-13–D-15)

| Prompt | Args (all optional) | Must include |
|--------|---------------------|--------------|
| `deploy` | `instance?`, `uuid?`, `force?` | Numbered steps: resolve target → `application.deploy` → monitor via `deployment.watch` (forward-ref Phase 21) with fallback `deployment.get` / status polling until watch exists (D-11) |
| `diagnose` | `instance?`, `uuid?` | Steps covering app/server/scan paths via `diagnose` tool actions; concrete `action` names |
| `new-project` | `instance?`, `name?`, `server_uuid?` | Setup/onboarding steps using existing project/environment/server tools; soft manifest write guidance; no recipe execution (Phase 20 deferred) |
| `incident` | `instance?`, `uuid?`, `project_uuid?` | Emergency/redeploy triage steps; prefer diagnose → logs → restart/deploy; never invent stub tools |

**Prompt depth:** Parameterized numbered steps with concrete `tool` + `action` calls. Not bare checklists. Not long playbooks (Phase 22).

**Prompt names:** Exact only — `deploy`, `diagnose`, `new-project`, `incident`. Reject namespaced variants (D-13).

### Discretion locks (researcher)

| Topic | Decision |
|-------|----------|
| Catalog wording | Hand-authored per tool; keep scannable Mission Control tone; prefer shortest accurate param names |
| `meta` / `docs` tools | If they already lack `oneOf` and expose params, keep structure; still add Safety footer if mutation/routing params exist; skip fake action catalogs when no action router |
| Prompt voice | Imperative steps (“Call…”, “Then…”) — agent as operator |
| Wow+Clarity | One signature pattern only (catalog + Safety footer); rest calm |

## Interaction Contract (MCP / agent)

| Interaction | Contract |
|-------------|----------|
| Schema inspection (high-frequency) | Flat Zod → JSON Schema with visible top-level properties; no decorative noise in descriptions |
| Action routing | Call shape remains `{ action, …fields }` (D-03) |
| Validation failure | Strict — wrong/missing fields → `COOLIFY_VALIDATION_ERROR` + `recoveryHints`; never silently drop extras (D-04) |
| Prompt open without args | Succeeds; body includes soft manifest/ask-user guidance (D-10, D-14) |
| Prompt vs tools | Prompts orchestrate; tools stay atomic (Pitfall 9) |
| Animation | N/A — never animate agent/tool flows (emil-design-eng) |
